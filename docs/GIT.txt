Git is a distributed version control system — software installed on your machine that tracks changes, manages branches, and handles operations like pull, push, fetch, commit locally. GitHub is a cloud hosting platform for Git repositories (not just "a repository" itself). GitHub stores your remote repository and adds collaboration features on top of Git — pull requests, code reviews, issues, actions, teams, permissions. You use Git locally to manage code, GitHub remotely to share and collaborate. Other platforms like GitLab, Bitbucket do the same thing — they're just different hosts for Git repositories. The key distinction: Git = the tool/protocol, GitHub = the platform/service that hosts Git repos.

Git Advantages: Comprehensive Study Notes
Git is a distributed version control system, fundamentally different from earlier centralized systems like SVN, PVCS, TFS, and Visual SourceSafe. The core advantage of Git is that every developer has a complete copy of the entire repository locally on their machine, including all history, all branches, and all commits. This distributed nature enables workflows and capabilities that were impossible with centralized systems.
One of the most powerful advantages of Git's distributed nature is the ability to create local branches that are invisible to everyone else. In a centralized system like SVN, branching is a server operation. When you create a branch in SVN, you are executing a command that communicates with the central server and creates a new directory path in the central repository. Everyone immediately sees this branch because it exists on the server. The branch is visible to all developers, creating clutter and making experimentation visible to the team.
In Git, branches are lightweight local constructs until you explicitly push them to a shared server. When you create a branch in Git, it exists only on your machine in your local .git directory. The central server like GitHub, GitLab, or Azure DevOps has no knowledge that this branch exists. You can create a branch, make multiple commits to it, experiment with different code approaches, rebase it to clean up history, delete it entirely, or modify it in any way — all completely invisible to everyone else. Only when you run the command to push the branch to the server does anyone else learn about it. This makes experimentation free and encourages developers to try different solutions without worrying about cluttering the shared repository or making their work visible before it is ready.
Creating local branches in Git can be done through multiple interfaces. Using the command line, you create a local branch by running git branch branch-name which creates a new branch based on your current branch, or you can use git checkout -b branch-name which creates the branch and switches to it in one command. These commands are entirely local operations. Nothing touches the server. You can run git branch to list all your local branches. These branches exist in your local repository only.
Using Visual Studio, the process is equally straightforward. You go to the Git menu or Team Explorer, select New Branch or Create Branch, enter the branch name, select which branch you want to base it on, and click Create. This creates the branch locally on your machine. In Visual Studio's Team Explorer, when you look at the Branches section, you will see two parts: Local branches and Remote branches. The local branches are the branches that exist only on your machine and have never been pushed to the server. Remote branches are the branches that exist on the server like Azure DevOps.
When you work on a local branch in Visual Studio, making commits, you are committing to your local repository. These commits stay on your machine. Your teammates see nothing. You can make ten commits, realize the approach is wrong, and delete the entire branch locally without anyone knowing you ever tried that approach. This is the freedom that Git provides. With SVN, if you tried an approach on a server branch and then decided to delete it, everyone would have seen the branch, wondered why it was deleted, and questioned your decision-making.
The interesting limitation is that you cannot create local branches in Azure Repos web interface or any other web-based Git hosting platform. When you use the web interface of GitHub, GitLab, Azure Repos, or Bitbucket and create a branch, that branch is created on the server immediately and becomes visible to everyone. This is because web interfaces have no concept of a local repository. They only interact with the server repository. A branch created in the web interface is a server-side operation, similar to how SVN worked. The web interface is a tool for managing and viewing the server repository, not for local development.
This is why developers use local tools like Visual Studio, VS Code, Git command line, or other IDE integrations for local development. These local tools understand the concept of a local repository. They can create branches locally, commit locally, and manage your local work without touching the server. The web interface is useful for viewing branches that already exist on the server, creating pull requests, reviewing code, and managing shared branches, but it is not the place to do local experimentation.
The practical workflow combines both approaches. You use Visual Studio or the command line to create local branches for experimentation. You try different approaches, each in a separate local branch. You can run git branch -a from the command line to see all your local branches (without the remote prefix) and all remote branches on the server (with the origin/ prefix). You compare your different approaches, run tests, verify the solution works correctly. Once you are satisfied, you use Visual Studio by going to Git → Push and clicking Publish Branch, or from the command line you run git push origin branch-name. This pushes your local branch to the server. Now the branch appears in Azure Repos, GitHub, or GitLab. Now your teammates can see it. Now it becomes part of the shared repository.
The ability to create invisible local branches is what makes Git's experimentation so cheap. In SVN, creating a branch was a visible, heavyweight operation that everyone saw. In Git, creating a branch is free and invisible. You can create ten branches, experiment with five different solutions, delete the bad ones, and push only the good one. The repository stays clean because only successful experiments make it to the server. With SVN, all five experimental branches would clutter the central repository, and other developers would see all your failed attempts.
This distributed branching capability combines with other Git advantages to create a superior development experience. Because branches are local and cheap, developers branch frequently. Because merging in Git is sophisticated and understands branch history, merging is less painful than SVN. Because you can rebase your local commits before pushing, the shared repository can maintain a clean linear history. Because you can work offline on local branches and commits, you can continue being productive without a network connection. You make commits locally, and when you reconnect to the internet, you push them to the server.
The performance advantage is also significant. Creating a branch in Git is instant because Git is just creating a pointer to a commit in your local repository. Creating a branch in SVN requires copying the entire codebase on the server, which takes time proportional to the size of your repository. Switching between branches in Git is instant because it is just updating your working directory to match the commit history. Switching branches in SVN requires fetching the branch contents from the server.
Git's distributed nature with local branches also provides disaster recovery benefits. If the central server crashes and all data is lost, you can restore the repository from any developer's local copy because they have the entire history. With SVN, if the central server crashed, the repository was lost unless you had backups. Every developer's local copy in SVN was just a snapshot of one version, not a complete repository.
The learning curve for understanding local branches in Git is steeper than understanding SVN branches because Git introduces the concept of local branches separate from remote branches. Developers need to understand that a branch they see locally might not exist on the server, and a branch on the server might not be on their local machine. They need to understand that pushing is an explicit operation that makes local work visible. However, once this is understood, Git's model is actually more intuitive because it matches how developers think: experiment locally, push when ready, share with the team.
The workflow with local branches also encourages better development practices. Because branches are cheap, developers branch for every feature or bugfix. This isolation prevents changes from interfering with each other. Because local work is invisible, developers feel free to commit frequently, creating a detailed history of how they arrived at their solution. When they push to the server, they can rebase or squash their commits to present a clean history to the team. This allows individual developers to have one style of committing for their own benefit, while the shared repository has a different, cleaner style that benefits the entire team.
Using Visual Studio as your primary development tool with Git is excellent because Visual Studio understands Git deeply. Creating branches, switching branches, committing, pushing, pulling, merging — all can be done through the GUI without touching the command line. Team Explorer shows your local branches, remote branches, and unpushed commits clearly. The Sync section shows your unpushed commits that are ready to push when you are ready to make them visible. The Branches section clearly distinguishes local from remote, making it obvious what is local and invisible versus what is on the server and visible.
The command line gives you additional power if you need it. git branch lists local branches. git branch -r lists remote branches. git branch -a lists all branches. git checkout branch-name switches to a local branch. git checkout -b branch-name creates and switches to a new local branch. git push origin branch-name pushes a local branch to the server, making it visible. git branch -d branch-name deletes a local branch if you have decided you do not need it. All of these are local operations or intentional synchronization with the server.
The distinction between local and remote is critical for understanding Git's power. Local branches enable invisible experimentation. Remote branches are the shared reality that the team sees. Most developers never need to touch the command line if Visual Studio's Git integration is sufficient, but understanding these commands helps understand what is happening under the hood. Visual Studio's Publish Branch button is doing git push origin branch-name behind the scenes. Visual Studio's Create Branch is doing git branch and git checkout behind the scenes.
In summary, Git's local branching capability is a transformative advantage over centralized systems. The ability to create invisible branches changes how developers work. Experimentation becomes free and risk-free. Developers can try multiple approaches, compare them locally, and share only the best one. The repository stays clean. Teammates see only finished work, not failed experiments. Combined with Git's other advantages like offline work, fast performance, sophisticated merging, and cryptographic integrity, local branching makes Git fundamentally superior to centralized systems like SVN for modern development. Whether using Visual Studio's GUI or command-line tools, the underlying capability is the same: Git lets you work locally, experiment freely, and share only when ready.

GitHub is a Cloud Service (SaaS), not a Cloud Provider.
Key distinction:

Cloud Providers = Azure, AWS, GCP — they provide infrastructure (virtual machines, databases, storage, networking, etc.)
Cloud Services/SaaS = GitHub, Slack, Salesforce, Office 365 — applications running on cloud infrastructure

GitHub is owned by Microsoft (acquired 2018) and is hosted on cloud infrastructure, but it's a SaaS platform — you access it via website/API as a service, not as infrastructure you rent.
GitHub.com's infrastructure uses multiple cloud providers (including AWS and Azure) for redundancy and performance, but the exact details aren't publicly disclosed. That's irrelevant to you — you just know GitHub is a cloud-based Git repository hosting service.
Analogy: Azure/AWS/GCP = the buildings. GitHub = the restaurant inside one of those buildings. You don't care which building the restaurant is in, you just use the restaurant's service.
