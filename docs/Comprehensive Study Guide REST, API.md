Comprehensive Study Guide: REST, APIs, and HTTP Methods
REST stands for Representational State Transfer, an architectural style created by Roy Fielding in his 2000 PhD dissertation at UC Irvine. REST is not owned by anyone because it is an architectural style and set of principles, not a product or protocol. Fielding essentially codified what already worked well with HTTP, defining the proper way to use HTTP for building web services. REST is based on six core principles: client-server architecture, stateless communication, cacheability, uniform interface using resources and verbs, layered systems, and optional code on demand. Before REST was formally defined, developers were misusing HTTP, sending everything through POST regardless of the actual operation. REST came into the picture to establish guidelines on which HTTP method to use for which operation.
Before REST became the standard, the primary technology for building web services was ASMX, which stands for ASP.NET Web Services. ASMX was built entirely around SOAP, the Simple Object Access Protocol. SOAP is an XML-based protocol that is protocol-agnostic, meaning it can work over HTTP, HTTPS, SMTP, TCP, and many other protocols. However, ASMX specifically implemented SOAP over HTTP and focused exclusively on XML as the data format. An ASMX web service required creating a web service file with a .asmx extension and defining methods decorated with the WebMethod attribute. When you called a ASMX service, you would send SOAP-formatted XML in the request and receive SOAP-formatted XML in the response. The entire communication was verbose and heavy, wrapped in SOAP envelopes with XML declarations. ASMX came from the early 2000s and is now considered outdated and legacy technology, but it was the first mainstream way for .NET developers to build web services.
After ASMX came WCF, which stands for Windows Communication Foundation. WCF was released in 2006 and was designed to be far more flexible than ASMX. The major difference with WCF is that it is protocol-agnostic and binding-agnostic, meaning you can choose which protocol and communication style to use. WCF can use SOAP over HTTP, SOAP over TCP, SOAP over named pipes, REST over HTTP, and many other combinations. You define the binding through configuration, specifying whether you want wsHttpBinding for SOAP, webHttpBinding for REST, or netTcpBinding for TCP communication. While WCF technically supports REST and can return JSON, doing so requires complex XML configuration and feels forced because WCF was fundamentally designed with SOAP and enterprise complexity in mind. WCF expects you to define service contracts and operation contracts using interfaces and attributes, making it feel overly complex for simple REST APIs. WCF was designed for large enterprise systems where you might need SOAP for banking systems over TCP, but it became cumbersome when developers just wanted to build simple REST APIs for the web.
The complexity of WCF for building REST APIs led Microsoft to create ASP.NET Web API, released around 2012. Web API was built from the ground up with REST in mind. Instead of thinking about bindings and contracts, Web API thinks in terms of HTTP verbs and controllers. You create controllers that inherit from ControllerBase, decorate actions with HTTP verb attributes like HttpGet, HttpPost, HttpPut, HttpPatch, and HttpDelete, and the framework automatically handles routing based on HTTP conventions. Web API naturally returns JSON by default and treats HTTP semantics as first-class concepts. Configuration is minimal, just registering controllers in startup code. Web API became the modern standard for building REST APIs in .NET because it is lean, intuitive, and REST-native. It was purpose-built for the web, not adapted from enterprise systems, making it the right tool for building REST APIs.
REST uses HTTP exclusively because it was designed around HTTP's model of stateless, cacheable, request-response communication. HTTP provides the fundamental mechanisms that REST leverages: status codes, headers, caching directives, and request methods. The HTTP methods are GET, POST, PUT, PATCH, DELETE, HEAD, OPTIONS, and others, but the main ones used in REST are the first five. REST defines semantic meaning for each method based on what operation you are performing on a resource.
GET is used to retrieve data from the server. When you send a GET request to a resource, you are asking the server to return that resource without modifying it. GET requests are safe and idempotent, meaning you can call them multiple times and get the same result every time without side effects. GET requests use query strings in the URL to pass parameters, such as GET /api/users?page=1&limit=10. GET should never have a body in the request because it is meant for retrieval only. The response contains the requested data. GET requests are cached by default by browsers, proxies, and CDNs. This caching is a huge performance advantage because subsequent requests for the same resource can be served from cache without hitting the server again.
POST is used for creating new resources or performing actions. When you send a POST request to an endpoint, you are asking the server to create something new or perform some operation. POST requests include a body containing the data needed for the operation. POST can also be used for searches or complex operations where you need to send structured data that is too complex for query strings. However, POST is NOT idempotent, meaning if you call POST multiple times, it performs the action multiple times. If you POST to create a user and the network drops, your client might retry, and the server processes the creation again, creating a duplicate user. POST requests are not cached by default.
PUT is used to update or replace a resource entirely. When you send a PUT request to a specific resource URL, you are saying "replace this entire resource with what I am sending." You must include all fields of the resource in the PUT request body. If you send only some fields, the other fields become null or deleted. PUT is idempotent, meaning calling PUT multiple times with the same data produces the same result. If you PUT to update a user and the network drops, the client can safely retry because calling PUT again with the same data just sets the resource to the same state. PUT tells the server and any caches to invalidate that resource.
PATCH is used for partial updates to a resource. When you send a PATCH request, you send only the fields you want to change, and the server updates only those fields, leaving other fields untouched. PATCH is also idempotent because updating the same fields to the same values multiple times produces the same result. PATCH is more practical than PUT for most APIs because clients typically only know the fields they want to change, not the entire resource structure.
DELETE is used to remove a resource. When you send a DELETE request to a resource URL, you are asking the server to delete that resource. DELETE is idempotent because deleting a resource multiple times results in the resource being deleted, and subsequent delete requests return a 404 Not Found, which is still a consistent result.
Now, why can't you use POST for GET, PATCH, PUT, or DELETE operations? The first reason is idempotency and network resilience. When networks fail, clients and load balancers automatically retry requests. If you use POST for updating a user, and the network drops after the update is processed but before the response reaches the client, the retry will perform the update again, causing data corruption. With PUT or PATCH, retries produce the same result every time because these methods are idempotent. Using POST for deletion is equally problematic. If a POST delete request fails and retries, the system might attempt deletion twice, causing errors or unexpected behavior. Using DELETE means that retrying a failed delete request is safe because the resource ends up in the same state: deleted.
The second reason is HTTP caching. GET requests are cached by default by browsers, proxies, and CDNs. If you use GET to fetch data, the response is cached, and the next request for the same resource returns the cached response without hitting the server, making your API much faster. POST, PUT, PATCH, and DELETE requests are not cached by default because they are state-modifying operations. If you used POST for a retrieval operation just because you wanted to send complex data in the body, you would lose all caching benefits and hit the server every time, degrading performance.
The third reason is semantic clarity and REST standards. When developers see GET, they expect retrieval. When they see POST, they expect creation or action. When they see PUT or PATCH, they expect updates. When they see DELETE, they expect deletion. Using the correct method communicates intent clearly, making APIs easier to understand and maintain. Following standards also means tools, frameworks, and developers around the world understand your API correctly.
The fourth reason is distributed systems and load balancers. When requests pass through multiple systems, each with retry logic, using non-idempotent methods for state-modifying operations causes cascading issues. A load balancer might route your first request to Server A, which processes it. If the response is lost, the load balancer retries and routes to Server B. With POST, both servers process the operation, creating duplicates. With PUT or PATCH, both servers set the resource to the same state, which is safe. In microservices architectures, this becomes critical for data integrity.
Why can't you use PUT and PATCH interchangeably? PUT is for replacing the entire resource, while PATCH is for partial updates. If a client sends a PUT request with only some fields, expecting them to be updated, the server replaces the entire resource with just those fields, nullifying all other data. PATCH is safer because it updates only the provided fields and preserves the rest.
The advantages of caching are significant. Caching reduces server load because repeated requests for the same resource are served from cache instead of hitting the server. Caching improves performance because cached responses are served instantly from the browser, proxy, or CDN without network latency. Caching reduces bandwidth usage because fewer requests reach the server. GET requests are cached by default because they are safe and idempotent. A browser automatically caches GET responses and serves them for subsequent identical requests. CDNs and proxies also cache GET responses, allowing content to be served from locations closer to users.
Beyond REST, newer technologies have emerged for specific use cases. GraphQL is a query language created by Facebook that allows clients to request exactly the fields they need from an API endpoint. Instead of calling GET /api/users/123 and receiving all user fields, you send a GraphQL query asking for only name and email. This reduces over-fetching and network bandwidth. GraphQL is gaining adoption in companies building frontend-heavy applications where flexibility in querying is valuable.
gRPC is a framework created by Google that uses HTTP/2 and Protocol Buffers instead of HTTP/1.1 and JSON. gRPC is binary-based instead of text-based, making it much faster and more efficient. gRPC is ideal for internal microservices communication where performance is critical, but it is not as widely adopted for public APIs as REST because it requires special clients and tooling.
WebSockets provide bidirectional, persistent connections between client and server, enabling real-time communication. Unlike HTTP, which is request-response based, WebSockets allow the server to send data to the client at any time without the client requesting first. WebSockets are used for chat applications, live notifications, collaborative tools, and any application requiring real-time updates.
Despite these alternatives, REST remains the dominant standard for most APIs and web services. REST is simple, well-understood, and has mature tooling across all platforms. REST naturally leverages HTTP caching and other HTTP infrastructure. REST is flexible enough for most use cases. REST is the default choice for building web APIs, and alternatives like gRPC and GraphQL are chosen only when their specific advantages are needed. Understanding REST, HTTP methods, and when to use each method is fundamental knowledge for any software developer building modern applications.
